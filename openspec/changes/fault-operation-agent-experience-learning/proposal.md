## Why

故障运维 Agent 当前主要依赖固定系统提示与（规划中的）SOP 向量召回做「参考定位」，缺少把**已解决会话**转化为可复用知识的闭环；同类故障反复出现时，模型无法系统性累积团队经验，导致「每次都像第一次排查」。需要在不破坏现有问答与 SSE 契约的前提下，引入**可审计、可治理**的长期学习与检索增强路径，使 Agent 随使用次数提升对相关域的命中率与步骤质量。

## What Changes

- 定义「运维经验」从**会话结束/用户确认解决**到**可检索资产**（结构化片段或 SOP 草稿/补丁）的采集与准入规则（含人工审核可选阶段）。
- 在故障运维链路中增加**经验检索**步骤：与现有 SOP 召回并列或融合排序（如 RRF），向 Agent 注入「历史同类已验证步骤」上下文。
- 持久化与隔离：按租户/用户/会话边界存储元数据，支持过期、版本与禁用（合规与纠错）。
- 观测与回滚：记录引用了哪条经验、置信度与反馈，支持关闭写入或仅读模式。
- **不**在本变更中要求替换现有 LLM/MCP 协议；**不**默认自动覆盖人工维护的权威 SOP（需显式策略才可晋升）。

## Capabilities

### New Capabilities

- `fault-operation-experience-learning`：故障运维场景下的经验采集、存储、检索注入与治理（与 PRD 中 SOP 召回设计对齐，补齐「用后沉淀」规格）。

### Modified Capabilities

- （无）当前 `openspec/specs/` 下无专用于故障运维的基线规格；本变更以新增能力规格为主。若后续实现将经验写入与 `knowledge-base` 共用同一 Qdrant 集合或复用入库 API，再在实施阶段评估是否对 `knowledge-base` 规格追加 delta。

## Impact

- **后端**：`backend/agent/fault_operation_agent.py`（注入检索上下文、工具或中间件）、`backend/services/qa_service.py`（故障流编排触点）、可能新增 `services/` 经验服务、`model/` 表或向量集合、`config/env.py` 开关与配额。
- **数据**：MySQL 元数据（经验条目、状态、来源会话）与/或 Qdrant 向量（与现有 SOP 索引策略一致时复用基础设施）。
- **API**：若需独立管理端点，走 `api/*` + `ResponseUtil`；若仅内部使用可无新对外 REST。
- **前端**：可选（管理台开关、反馈「已解决」）；首版可纯后端 + 聊天内隐式触发。
- **依赖**：与现有 DashScope、MySQL、（可选）Qdrant 一致；不引入第二套并行 Agent 框架。
